## 一、设计目标

为 4 岁幼儿设计一套古诗启蒙测试系统，核心目标是：

1. **感知韵律之美** — 通过节奏感知题感受古诗的韵律
2. **建立语言亲近感** — 通过图片和游戏化方式让幼儿亲近古诗
3. **游戏化测评** — 全程图片化、游戏化，不考试、不排名

## 二、迭代历程

### 版本 1.0 — 静态测试题（23:18）

**实现方式**：纯 Markdown 模板，人工设计题目

**题目类型**：
- 听音辨图题（老师念诗，幼儿选图）
- 看图说诗题（看图背诵）
- 诗句排序题（给散图排序）
- 节奏感知题（拍手/跟读）
- 意象指认题（指认诗中形象）
- 情感感知题（感受诗歌意境）

**问题**：题目固定，用户无法交互，只能看答案

---

### 版本 2.0 — AI 交互引擎（23:33）

**新增**：Python 脚本 quiz_engine.py，实现 AI 自动出题 + 交互验证

**实现方式**：
```bash
python3 quiz_engine.py 咏鹅  # 交互模式
python3 quiz_engine.py 春晓 --quiet --answers AAB  # 静默模式
```

**题目类型**：2选1（A/B）选择题

**问题**：
- 正确答案固定在 A，无法真正测试
- 需要用户在终端输入，体验割裂

---

### 版本 3.0 — 4选1随机打乱（23:45）

**改进**：
1. 选项从 2 个扩展到 4 个（A/B/C/D）
2. 正确答案位置每次随机，不再固定 A
3. 每道题目的选项顺序也随机打乱

**新增干扰项**：
| 古诗 | 干扰项示例 |
|------|-----------|
| 咏鹅 | C=清水、D=大海 |
| 春晓 | C=鸡鸣、D=蛙声 |
| 悯农 | C=美味、D=新鲜 |

**问题**：交互仍在终端，用户体验仍不流畅

---

### 版本 4.0 — 追问模式（00:04）

**核心改进**：AI 主动调用脚本 + 展示题目给用户，用户在聊天里回复

**工作原理**：
```
用户："古诗测试"
         ↓
    AI调用脚本 → 出第1题 → 脚本退出（状态保存）
         ↓
    用户在聊天里回复 "D"
         ↓
    AI调用脚本 --resume D → 判定对错 → 出下一题
         ↓
    循环……全部答完 → 显示汇总结果
```

**状态持久化**：使用 /tmp/poetry-quiz-state.json 保存进度

**问题**：
- webchat 不支持按钮点击，用户需手动打字
- 需要明确告知用户"直接输入字母即可"

### quiz_engine.py 核心逻辑

```python
# 两种运行模式：
# 1. 默认（追问模式）：出题后立即退出，等待用户回复
# 2. --quiet（静默模式）：跑完全部题目，输出 JSON 结果

# 状态持久化：
# 使用 /tmp/poetry-quiz-state.json 保存当前进度
# 支持跨调用连续答题

# JSON 输出格式（供 AI 解析）：
{
  "type": "quiz",           # started | quiz | wrong | correct | done
  "poem": "咏鹅",
  "current": {
    "q_num": 1,
    "question": "白毛浮（      ），红掌拨清波。",
    "options": {"A": "绿水", "B": "蓝天", "C": "清水", "D": "大海"},
    "correct_key": "A",     # 正确答案位置（随机）
    "_correct_value": "绿水",
    "hint": "提示：白鹅喜欢在绿绿的水里游泳~"
  }
}
```

## 四、题库设计

### 内置古诗（5首）

| 古诗 | 朝代/作者 | 题数 | 题目类型 |
|------|-----------|------|---------|
| 《咏鹅》 | 骆宾王（唐） | 4题 | 选择题 |
| 《春晓》 | 孟浩然（唐） | 3题 | 选择题 |
| 《悯农》 | 李绅（唐） | 3题 | 选择题 |
| 《静夜思》 | 李白（唐） | 3题 | 选择题 |
| 《登鹳雀楼》 | 王之涣（唐） | 2题 | 选择题 |

### 题目设计原则

| 成人测试 | 幼儿版本 |
|---------|---------|
| 诗句默写 | 看图背诵（配图提示） |
| 诗意理解 | 听音辨图（配哪幅画？） |
| 中心思想 | 情感感知（开心还是安静？） |
| 作者朝代 | ~~删掉~~（不考） |
| 字词解释 | ~~删掉~~（不考） |

### 答题规则

- 每题最多 3 次尝试
- 答错后显示提示（不影响分数）
- 3 次都错 → 平静告知正确答案，继续下一题
- 全程无分数压力，不批评，不排名

## 五、AI 使用指南

### 用户操作流程

```
1. 用户："古诗测试" / "来测试《咏鹅》"
         ↓
   AI 调用脚本（无参数）→ 输出第1题 + JSON 状态
         ↓
   AI 渲染题目给用户（Markdown 格式）
         ↓
   用户回复字母（如 "D"）或说"提示"
         ↓
   AI 调用脚本 --resume D → 处理答案 → 输出下一题
         ↓
   循环……全部答完 → 显示汇总结果
```

### 状态解读

| JSON type | AI 动作 |
|-----------|---------|
| `started` | 显示古诗标题和说明 |
| `quiz` | 渲染当前题目 |
| `wrong` + `remaining > 0` | 鼓励 + 显示提示 |
| `wrong` + `remaining = 0` | 告知正确答案 + 下一题 |
| `correct` | 表扬 + 下一题 |
| `done` | 显示完整成绩单 |

## 六、设计决策记录

### 决策 1：为什么用追问模式而非独立程序？

**方案 A**：让用户在终端运行 `python3 quiz_engine.py 咏鹅`
- ❌ 用户体验割裂，需切换窗口
- ❌ 需要安装 Python 环境

**方案 B**：AI 主动调用脚本 + 渲染题目给用户
- ✅ 用户全程在聊天窗口操作
- ✅ AI 控制流程，可随时介入

**结论**：采用方案 B（追问模式）

---

### 决策 2：为什么从 2 选 1 改为 4 选 1？

**原因**：
- 2 选 1 随机蒙对概率高达 50%，无法真实测试
- 4 选 1 随机蒙对概率降至 25%
- 加入干扰项（清水/大海/蓝天）增加区分度

---

### 决策 3：为什么正确答案位置要随机？

**原因**：
- 防止用户"背答案"（上次选 A 对，这次还选 A）
- 每次都需要真正理解题目才能答对
- 增加测试的公平性和有效性

---

### 决策 4：为什么不直接用按钮点击？

**原因**：
- OpenClaw webchat 当前不支持 Markdown 按钮交互
- 改为"直接打字回复"——用户体验同样简单
- AI 自动识别用户输入的字母（A/B/C/D）
